How it works

HTTP 402. USDC underneath.

Standard MPP flow on the outside. Exact USDC transfer verification on Base on the inside. The upstream API only sees a normal forwarded request.

1. Agent requests

Client calls the proxied endpoint with no payment credential.

GET /p/:id/resource

2. Proxy challenges

PIMPP returns 402 Payment Required and a strict MPP challenge.

WWW-Authenticate: Payment ...

3. Client pays

The client sends a USDC transfer on Base and retries with a payment credential.

payload: { txid }

4. Proxy verifies

PIMPP verifies amount, recipient, challenge state, and replay protection, then forwards upstream.

200 OK + Payment-Receipt
Integrate

Register an API or pay any compatible endpoint.

The product should integrate the way zimppy integrates: hosted proxy, reusable payment package, and a CLI path for immediate use.

Server / Provider

Register a base endpoint and receive a paid proxy URL.

POST /register
{
  "originUrl": "https://api.example.com/v1",
  "priceUsdc": "0.01",
  "destinationWallet": "0x..."
}

→ { "proxiedUrl": "https://pimpp.fun/p/abc123" }

Client / SDK

Use mppx with the reusable USDC-on-Base method.

import { Mppx } from 'mppx/client'
import { usdcBaseClient } from '@pimpp/usdc-base'

const mppx = Mppx.create({
  methods: [usdcBaseClient({ privateKey, rpcUrl })],
  polyfill: false,
})

const res = await mppx.fetch(url)
